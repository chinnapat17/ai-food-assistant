## 1. การทดลองและลงมือปฏิบัติจริง (Implementation Reality)

ในการพัฒนาโครงงานระบบแนะนำร้านอาหารกลุ่มใหญ่นี้ สิ่งที่เกิดขึ้นจริงในขั้นตอนปฏิบัติคือการนำเอาแนวคิดโมเดล **AI Workflow แบบ End-to-End** มาประยุกต์ใช้ในสถานการณ์จริง โดยการจำลองการ Scrape ข้อมูลดิบจำนวน 100 ร้านค้าในย่าน Siam, Ari, Thonglor, Asoke และ Phrom Phong ผ่านแพลตฟอร์ม Apify แล้วบันทึกข้อมูลแบบเรียลไทม์ลง Google Sheets จากนั้นใช้ n8n Automation และ Gemini 3.5 Flash ในการจัดหมวดหมู่ วิเคราะห์ Trade-off และให้คะแนนเต็ม 100 ตามเกณฑ์ประเมิน ก่อนจะส่งมอบข้อมูลมานำเสนอบนหน้า Web Dashboard ดีไซน์หรูหราแบบ Dark Glassmorphic ที่ทำงานด้วยระบบ Google Sheets Live Sync

การทำงานจริงชี้ให้เห็นว่า ข้อมูลดิบจาก Scraping มีความกระจัดกระจาย ขาดมาตรฐานความสะอาดของข้อมูล (Data Cleanliness) และการทำความสะอาดข้อมูลด้วยโค้ดแบบดั้งเดิม (Vanilla JS/Regex) มีข้อจำกัดสูงเนื่องจากความหลากหลายในการเขียนชื่อประเภทอาหารและที่อยู่ การนำ AI (Gemini 3.5 Flash) เข้ามาประเมินร่วมในไปป์ไลน์ทำงานได้มีประสิทธิภาพมาก สามารถสกัดข้อมูลเชิงลึกและระบุจุดเด่นจุดด้อยออกมาในรูปแบบภาษาธรรมชาติที่เป็นมนุษย์ได้ทันที

## 2. จุดที่ทำได้ดีและประสบความสำเร็จ (Strengths & Successes)

*   **การนำเกณฑ์ Rubric 100 คะแนนเต็มมาใช้ประเมิน:** ทำให้โมเดล AI มีความเที่ยงตรงสูงในการจัดอันดับร้าน โดยแบ่งเกณฑ์อย่างชัดเจน ได้แก่ ความพึงพอใจรีวิว (25 คะแนน) ความเหมาะสมสำหรับกลุ่มใหญ่ (20 คะแนน) ความเหมาะสมราคา (15 คะแนน) การเดินทาง (15 คะแนน) ความครบถ้วนข้อมูล (15 คะแนน) และความพิเศษเอกลักษณ์ (10 คะแนน)
*   **ระบบความปลอดภัยและการมีมนุษย์ในลูป (Human-in-the-loop):** สามารถคัดกรองร้านที่มีรีวิวต่ำกว่า 150 ครั้งเข้าสู่ระบบ "Human Review Queue" เพื่อป้องกันไม่ให้ข้อมูลดิบที่เชื่อถือไม่ได้มาทำลายน้ำหนักคะแนนรวม และปุ่ม Approve/Reject ก็ทำงานได้อย่างเสถียรและจำลองการเก็บข้อมูลในระดับ Browser (localStorage) ได้อย่างสมบูรณ์
*   **การเพิ่มประสิทธิภาพ Mobile UX/UI:** การปรับเปลี่ยนโครงสร้างตารางปกติ (Tables) ให้กลายเป็นแบบแอปพลิเคชันการ์ดแนวตั้ง (Card Layouts) บนหน้าจอมือถือ ทำให้ข้อมูลตารางที่เคยล้นจอและอ่านยากมากบนสมาร์ตโฟน กลับมาอ่านง่าย เป็นสัดส่วน และไม่กระจัดกระจาย พร้อมการปรับแก้สีสันเฉพาะของอันดับ 1, 2, และ 3 (Gold, Silver, Bronze) บนทั้งสองส่วนของการ์ดแนะนำ ทำให้มิติภาพของ Dashboard ดูพรีเมียมขึ้นทันตา

## 3. ปัญหาและอุปสรรคที่พบ (Issues & Challenges)

*   **ปัญหา Encoding ในขั้นตอน Scraping:** ในช่วงแรกที่พยายามทดสอบดึงข้อมูลดิบในเขตกรุงเทพฯ เกิดปัญหาการระบุพิกัดตำแหน่งผิดพลาดเนื่องจาก encoding ภาษาไทยเพี้ยนในขั้นตอนการส่ง JSON เบื้องหลัง ส่งผลให้บอทวิ่งไปค้นหาสถานศึกษาหรือสถานที่ราชการแทนที่จะเป็นร้านอาหาร ปัญหานี้แก้ได้โดยสร้างไฟล์ตั้งค่าในรูปแบบ UTF-8 ที่ไม่มี BOM (Byte Order Mark) และเรียกใช้ผ่านคำสั่ง curl โดยตรง
*   **ปัญหาข้อมูลซ้ำซ้อน (Data Duplication):** เมื่อเกิดการแก้ไขข้อมูลในแถวเดิมของแท็บ Raw Data ระบบ n8n จะเข้าใจว่ามีข้อมูลร้านอาหารใหม่เข้ามา และทำการบันทึกข้อมูลแถวใหม่ต่อท้ายเสมอ ทำให้ร้านค้าเดิมเกิดการอัปโหลดซ้ำขึ้นไปในแท็บผลลัพธ์เป็นจำนวนมาก ส่งผลให้ระบบฐานข้อมูลเบื้องหลังเละเทะและคำนวณอันดับเพี้ยน
*   **ปัญหาด้านความปลอดภัยของเบราว์เซอร์ (CORS):** การพัฒนาหน้าเว็บโลคอลที่เปิดไฟล์ด้วย `file://` ติดปัญหา CORS เมื่อดึงข้อมูลภายนอกจาก Google Sheets API ทำให้ดึงข้อมูลไม่สำเร็จจนกระทั่งนำไป Deploy ขึ้นระบบคลาวด์ Vercel

## 4. กรณีที่ AI แนะนำผิดพลาดและการแก้ไข (AI Hallucinations & Fixes)

### • การแนะนำใช้ JSON Stringify ในไปป์ไลน์ n8n
*   **รายละเอียด:** ตอนแรก AI แนะนำให้พิมพ์ตัวแปรดิบลงไปใน Request Body ของ Gemini Node ตรงๆ (เช่นการอ้างอิงฟิลด์ชื่อร้านอาหารดื้อๆ) ซึ่งเมื่อรันไปป์ไลน์จริง เมื่อพบร้านอาหารที่มีวงเล็บหรือเครื่องหมายอัญประกาศในชื่อ เช่น *ร้านหมูกระทะพรีเมียมเตาถ่าน (เทสระบบ)* โครงสร้าง JSON ของตัวแปรร่างที่ส่งไปยัง Gemini จะพังทันที ส่งผลให้ n8n แจ้งเตือนข้อผิดพลาด Bad Request (400) ตลอดเวลา
*   **การแก้ไข:** ผมเลิกใช้รูปแบบการเขียนโค้ดดิบแบบที่ AI แนะนำ แล้วเปลี่ยนมาเขียนนิพจน์ (Expression) ใน n8n โดยใช้ฟังก์ชันทางจาวาสคริปต์ `JSON.stringify({...})` ในการแปลง Request Body ทั้งอ็อบเจกต์ ซึ่งวิธีนี้จะช่วยหลบหลีกอักขระพิเศษ (Escape characters) และครอบเครื่องหมายคำพูดให้โดยอัตโนมัติ ทำให้ไปป์ไลน์ผ่านฉลุยและรองรับชื่อร้านอาหารได้ทุกกรณี

### • การแนะนำการเชื่อมต่อ Google Sheets แบบธรรมดา
*   **รายละเอียด:** AI แนะนำให้ใช้โหมด Write หรือ Append ธรรมดาเมื่อต้องการบันทึกข้อมูลของร้านที่ได้คะแนนแล้ว ซึ่งเป็นที่มาของการเกิดข้อมูลแถวใหม่ซ้ำๆ ทุกครั้งที่มีการแก้ไขเซลล์ดิบเดิมใน Google Sheets
*   **การแก้ไข:** ผมได้เปลี่ยนโหมดการบันทึกข้อมูลในโหนดสุดท้ายของ n8n จาก Append เป็นการใช้คุณสมบัติ **Upsert** (Update or Insert) และกำหนดให้ใช้ฟิลด์ ID ของร้านอาหาร (placeId) เป็นคีย์หลักในการระบุตำแหน่ง หากมีข้อมูล placeId เดิมอยู่แล้ว ระบบจะเข้าไปทับหรืออัปเดตเฉพาะค่าใหม่ในแถวเดิม แทนที่จะไปเพิ่มเป็นแถวใหม่ด้านล่าง ทำให้ข้อมูลในแท็บ Clean & Scored Data มีความสะอาด เรียบร้อย และไม่มีรายการซ้ำซ้อน

## บทสรุปเรียนรู้ (Conclusion Summary)

> โครงงานนี้สอนให้รู้ว่า การใช้ AI ทำงานในอุตสาหกรรมจริงไม่ได้ขึ้นอยู่กับความฉลาดของตัวแบบประมวลผลคำตอบ (Gemini) เพียงอย่างเดียว แต่ขึ้นอยู่กับการสร้าง **'Workflow และสถาปัตยกรรมข้อมูล'** ที่รัดกุม การตรวจสอบโครงสร้าง JSON ล่วงหน้า การใช้โหมดการบันทึกข้อมูลที่ถูกต้องอย่าง Upsert และการทดสอบระบบปลายทาง (Mobile Responsive และ Dynamic Theme) บนหน้าจอของอุปกรณ์จริง ล้วนเป็นปัจจัยหลักที่จะทำให้อุปกรณ์และขั้นตอนวิธีที่ซับซ้อน สามารถส่งมอบประโยชน์สูงสุดให้กับผู้ใช้ปลายทางได้อย่างแท้จริง
